# cxr 实时跟随模式设计

## 背景

当前 `cxr` 在不带参数时会读取最新的主代理 rollout，一次性输出筛选后的最近 100 条内容。这个默认行为适合回看历史，但不适合持续观察当前会话，因为命令执行完成后不会继续跟随新的 rollout 追加内容。

本次改动需要在不改变默认 `cxr` 行为的前提下，增加一个“实时更新最近 100 条或 n 条”的能力。

## 目标

- 保持 `cxr` 的现有默认行为完全不变
- 新增一个显式参数，启用实时跟随
- 实时模式启动时先输出最近 100 条或 `--count n` 条
- 后续当同一个 rollout 文件继续追加 JSONL 行时，持续打印新增提取项
- 优先选择性能更高的方案

## 方案比较

### 方案 A：周期性整量刷新

命令每隔固定时间重新扫描 `~/.codex/sessions`、重新选择最新 rollout、整文件读取并重算最近 `n` 条，再整屏刷新显示。

优点：
- 语义直接，始终对应“当前最新主会话”

缺点：
- 性能差，会重复扫描目录、重复读取和解析整文件
- 终端刷新噪音大，容易闪烁
- 如果最新会话切换，输出上下文会突然跳变

### 方案 B：增量文件跟随

命令启动时先执行一次现有选择与提取逻辑，然后锁定所选 rollout 文件，只跟随该文件的后续追加内容，按新增 JSONL 行增量解析并输出新增提取项。

优点：
- 性能更高，启动后只处理新增内容
- 终端体验稳定，像 `tail -f`
- 语义清晰，不会在输出过程中切换到别的会话

缺点：
- 需要处理半行写入和重复输出去重
- 默认不自动切换到新的最新会话

## 选型

采用方案 B。

理由：
- 用户明确要求“保持当前行为不动”，因此新能力应是显式模式，而不是改掉默认输出逻辑
- 当前实现以“整文件读取 + 提取 + slice(-count)”为主，增量跟随比周期性全量刷新更符合现有代码结构，也明显更省性能
- 跟随固定 rollout 文件的行为更稳定，适合作为实时观察工具

## 交互设计

- 新增 `--watch` 参数
- `cxr --watch`：
  - 先输出当前最新 100 条
  - 然后持续输出同一个 rollout 的新增内容
- `cxr --watch --count 20`：
  - 先输出当前最新 20 条
  - 然后持续输出新增内容

## 范围与约束

- 默认 `cxr` 行为不变
- `--watch` 不自动切换到新的最新会话
- 首版只支持终端实时输出，不和 `--save` 组合做特殊增强
- `--watch` 在 `--json`、`--include-tools`、`--include-mcp`、`--only` 等现有筛选语义下继续生效，输出规则与非 watch 模式一致

## 技术设计

### CLI 层

在 `src/cli.js` 中：
- 增加 `--watch` 参数解析
- help 文案补充 `--watch`
- 现有一次性模式保持原样
- 当 `--watch` 打开时：
  - 先按现有逻辑输出初始结果
  - 再进入文件跟随循环

### 跟随实现

在 `src/core.js` 中新增可复用能力：
- 读取 rollout 文件当前已有 entries
- 根据 options 提取当前匹配项
- 对新增文本维护 buffer，直到遇到完整换行才尝试 JSON 解析
- 对新解析出的 entries 重新执行提取逻辑，但只输出新增提取项

### 去重策略

启动时已经输出过最近 `n` 条，因此 watch 阶段只处理“文件新追加的 entries”。不对旧 entries 重跑输出，也不依赖 message 文本去重。

### 容错

- 新增行若暂时不是完整 JSON，一直缓存在尾部 buffer，等下次追加后再解析
- 若形成完整行但 JSON 非法，沿用现有报错风格并终止

## 测试策略

- 新增 CLI 级测试，验证 `--watch` 启动时先打印最近 `n` 条
- 验证 rollout 文件追加新 assistant 消息后，进程继续输出新增内容
- 验证 `--watch --count 2` 只输出初始最近 2 条，但后续仍能接着输出新增条目
- 保留现有默认模式测试，证明默认行为未变

## 不做的事

- 不实现自动切换到新会话
- 不实现整屏刷新型 live 模式
- 不引入额外依赖
